## Karabiner-Elements の設定ミス

今この雑記帳を書いていて気づいたが、先日に mac os tahoe にアップグレートして以降、Karabiner-Elements が意図通りに動作していなかったことに気づいた。
ESC を押したら入力ソースをデフォルトの半角英数字用のものに切り替えるようにしていた。
Vim でインサートモードで日本語入力している最中にノーマルモードに ESC 一発で戻りたい、という意図。

今使っている端末をキッティングしたときに、適当に ESC to en US という設定をどこからか持ってきて有効にしていたのだが、おそらく入力ソース名が変わったか何かで動かなくなっていたのかな。。

https://ke-complex-modifications.pqrs.org/#japanese からインストールできる「esc キーを押したときに、英数キーも送信する（vim 用）」というものを使うように変更。
こちらは、英数キーを追加発動させる、という設定であるため、入力ソース名に依存視せずに動作する。

```json
{
  "description": "escキーを押したときに、英数キーも送信する（vim用）",
  "manipulators": [
    {
      "from": { "key_code": "escape" },
      "to": [{ "key_code": "escape" }, { "key_code": "japanese_eisuu" }],
      "type": "basic"
    }
  ]
}
```

## Golang 環境構築

Go 言語のお勉強を行うために開発環境を作る。
といっても Claude に丸投げしただけだが。

> 1. Homebrew で Go 本体をインストール（go1.26.4、/Users/quramy/opt/homebrew/bin/go）
> 2. go install golang.org/x/tools/gopls@latest で Go 公式 LSP サーバー gopls（v0.22.0）をインストール
> 3. ~/.zshrc.local に export PATH="$HOME/go/bin:$PATH" を追記し、go install したバイナリをコマンドとして使えるように設定
> 4. ~/git/dotfiles/.vimrc の augroup golang に omnifunc=lsp#complete を追加
> 5. 同じく vimrc に、rust-analyzer と同形式で gopls の lsp#register_server ブロックを追加し、vim-lsp 経由で補完・定義ジャンプができるように設定
> 6. hello.go を作成し go run で実行確認、interactive シェルで go/gopls 両方の PATH 解決を確認
> 7. GOPATH（~/go）は Go 1.21+のデフォルトのため明示設定はせず、GOROOT も Homebrew 管理のため変更不要

昔は gofmt などのコマンドを利用する形で vim を設定していたようだったが、今回は gopls を提案されたので、公式 LSP ということもありそちらを利用することにした。

### 2 章

まずは 組み込み型の理解から。

整数は byte, unit8(符号なし 8bit 整数), int16, int32, int64 などがある。
精度指定なしの `int` は環境依存となり、64bit OS では int64、32bit OS では int32 となる。

浮動小数は float32, float64, さらに複素数型として、complex64, complex128 がある。複素数型を使うシーンは全く想像できないが。。

変数の宣言方法にいくつかの種類がある。

```go

    c := 0
}
```

関数内部であれば、 ゼロ値で型宣言したい場合を除き、 `c := 0` のような `:=` を使うのが一般的らしい。

## Makefile の .PHONY

久々に Makefile 書いたのだけど、そういえば .PHONY って何だっけ、と思ったので claude に聞いた。

make コマンドの成り立ちとして、作成される成果物をターゲットのラベルとする慣習があったわけで、 (多分 `main.o` のようなもの) その成果物のファイルが存在する場合、make はそのターゲットを作る必要がないと判断することによってビルド時間を削減する仕組みがある。

なので、以下のような部分は「 `fmt` というファイル名をチェックして、存在しなければ `go fmt ./...` を実行せよ」という意味になる。

```makefile
fmt:
    go fmt ./...
```

しかし、ここでの `fmt` はファイル名ではなく、ただのアクションに対するラベルなので、それを make に伝えるために .PHONY という特殊なターゲットを使うことで誤作動を防ぐよう慣例、ということの模様。
phony という単語の意味がそのまま紐づかないので、ちょっとわかりにくさがあるよなぁ。。。

## 初めての Go 言語 二日目

### 3 章

スライス周りの話:

- スライスは可変長配列として扱える型
- 現在の長さ (`len`) と容量 (`cap`) を持つ。 長さ <= 容量 である。
- `append` などの操作で、容量を超える場合は内部的に別のメモリ領域にもとのスライスの値がコピーされた上で、新しい容量を持つ領域が作成されている。
- サブスライス(スライスのスライス)は JavaScript の `Array.prototype.slice` とは違い、値(と容量)がを共有される。
  - このため、切り出し元の値を切り出した後から容易に破壊できてしまう

マップ周りの話:

マップに key が存在しない場合、値型に対するゼロ値が返却される。
key 自体の有無を識別するには、第 2 戻り値を利用する。

```go
x := map[string]int{
	"Hoge": 0,
	"Fuga": 1,
}

v, ok := x["Hoge"]
fmt.Println(v, ok) // 0, true

v, ok = x["Hogee"]
fmt.Println(v, ok) // 0, false
```

`value, ok := ...` のような書き方は Comma OK Idiom という名前で知られている。

ストラクト周りの話:

以下のような記法。Prisma とか GraphQL と近い。

```go
type user struct {
    id   string
    name string
}
```

すべてのフィールドが比較可能な型で構成された構造体について、インスタンス同士の `==` による比較は全フィールド同士の比較になる。

```go
u1 := user1{
	id:   "user01",
	name: "bob",
}

u2 := user1{
	id:   "user01",
	name: "bob",
}

fmt.Println(u1 == u2) // true
```

## TypeScript Native LSP と vim-lsp

以前からその内やらねばなーと思っていた、Native TypeScript LSP と tsserver の使い分けを Vim に仕込む件。

TypeScript v7 で LSP を起動するには `tsc --lsp --stdio` を実行するだけ。

`tsserver` があれば tsuquyomi, なければ vim-lsp となるように .vimrc を設定して試したのだが、vim-lsp 側が想定通りに動いてくれなかった。

原因は tsc --lsp がハンドシェイク直後に `client/registerCapability` を送信してくるが、これに vim-lsp が応答できていない、というもの。

https://github.com/Quramy/vim-lsp/commit/f4a3fed653655c44fd71202e66590777ebe1bb63 のように雑に空応答を返却すれば直るのだけど、 https://github.com/prabirshrestha/vim-lsp/issues/598 を読むと作者はどうもこの対応に納得しなさそうに見える。

> vim-lsp doesn't say it support client registrations which means the reason-language-server shouldn't send those message.

ref: https://github.com/prabirshrestha/vim-lsp/issues/598#issuecomment-566353191

vim-lsp は 「Client Registration に対応しているとは言っていないのだから、LSP サーバー側が "client/registerCapability" を送信してくるのが間違っている」という立場のようだ。

とりあえず、vim-lsp を fork することにしたものの、ちょっともやつく感じになった。

## `sibling-index()` 関数を用いた CSS アニメーションの例

https://master.dev/blog/in-n-out-animation-using-sibling-index/

要素の増減時に要素の index 値に応じて位置をアニメーションさせるという利用例が記載されていた。
`grid-area` で全要素が同じ場所に重ねられているところもポイント。

```html
<style>
  .container {
    display: grid;

    & * {
      grid-area: 1 / 1;
      width: 150px;
      height: 150px;
      transform: calc((sibling-index() - 1) * 160px);
      transition: transform 0.3s ease-in-out;
    }
  }
</style>

<ul class="container">
  <li>item</li>
  <li>item</li>
  <li>item</li>
  <li>item</li>
  <li>item</li>
</ul>
<button
  onclick="document.querySelector('.container > :nth-child(2)')?.remove()"
>
  Remove
</button>
```

`attr()` type 対応が住んでいれば、より柔軟な calc 式が書けそう。

## 初めての Go 言語 四日目

関数関係

関数も一級市民としてあつかえる、クロージャからレキシカルスコープの変数を束縛する、などは JavaScript と同じ感覚。

逆に JavaScript には無い概念は `defer` による遅延実行と関数引数の扱い。

`defer` は JavaScript に最近導入された Explicit Resource Management における `using` が近い(というか、 `using` が go の `defer` に影響受けてるはず)。
defer というキーワードのとおり、関数やメソッド、クロージャを遅延実行する。

関数引数については、基本的に値渡しとなる。
構造体型が引数であっても、そのコピーが作成されるため、構造体のメンバを関数内で変更したところで呼び出し元の変数は影響を受けない。
ただし、マップとスライスは別。
