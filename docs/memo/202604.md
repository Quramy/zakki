## TypeScript の --rewriteRelativeImportExtensions と Node.js の Strip Types

talt から Jest を剥がすという文脈で、移行先を Node.js Test Runner にしようとした。

すごい簡素化すると、以下のファイルに対して `node --test src/index.test.ts` を実行する、ということだ。

```ts
/* src/index.ts */

export function add(a: number, b: number): number {
  return a + b;
}
```

```ts
/* src/index.test.ts */

import { test } from "node:test";
import { add } from "./index";
// import { add } from "./index.js";
// import { add } from "./index.ts";

test("add", (t) => {
  t.assert.equal(add(1, 2), 3);
});
```

問題となるのが、 `import { add } from "./index";` の部分で、Node.js の ESM では拡張子が必要なため、これを `import { add } from "./index.js";` に書き換える必要があるが、これは tsc でトランスパイルした後の話なわけで、いわゆる `--strip-types` オプションで直接 Node.js から TypeScript を実行する場合は `from "./index.ts"` でないといけない。そのために tsc で `--rewriteRelativeImportExtensions` を有効にすることで、.ts 拡張子を Module Specifier として許容させる。

当たり前だが、テストファイル -> テスト対象 という依存関係だけでなく、テスト対象から先の依存モジュールについても ".js" や ".mjs" のままだと Strip Types では実行できないわけだから、それらもソースコード上は書き換えた上でビルド時に ".ts" を ".js" に戻す、という対応が必要となる。

talt 程度の規模だと、依存パッケージを極力減らしたいという側面で Node.js Test Runner を選んだが、以下の記事でも言及されているように `--rewriteRelativeImportExtensions` がやっていることを考えるとカジュアルに使い倒すものにはなり得ないな、という感覚。

https://zenn.dev/uhyo/articles/rewrite-relative-import-extensions-read-before-use#--rewriterelativeimportextensions-%E3%82%AA%E3%83%97%E3%82%B7%E3%83%A7%E3%83%B3%E3%82%92%E4%BD%BF%E3%81%86%E3%81%B9%E3%81%8D%E3%81%8B

## Dockerfile の COPY とワイルドカード

とあるレポジトリに含まれている Postgres 用の Dockerfile にて。

```
docker-compose.yml
postgres/
  Dockerfile
  init.sql
```

ファイルが上記のような構成になっていて、compose から postgres/Dockerfile をビルドする、というようなまぁよくあるパターンのはずなんだけど、どうにも init.sql がコンテナ内で走った痕跡がない。

```Dockerfile
FROM postgres:15.3

COPY *.sql /docker-entrypoint-initdb.d/
```

ワイルドカードではなく、ファイル名ベタ書きにしたら動作したんだが、何がダメだったのやら。

```
COPY init.sql /docker-entrypoint-initdb.d/
```
