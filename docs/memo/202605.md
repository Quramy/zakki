# 5/13

## TanStack が受けたサプライチェーンアタックのポストモーテムから学ぶ

https://tanstack.com/blog/npm-supply-chain-compromise-postmortem

詳細な時系列は上記に存在するが、5/12 日未明(JST) に、TanStack が所有する npm パッケージのいくつか(42 パッケージ, 合計 84 バージョン) がマルウェア付きで公開されていた（5/13 現在では npm から削除済み)。

### 攻撃の流れ

攻撃者が npm レジストリへ不正なコードを公開すまでに行った流れは以下:

1. GitHub Action の Cache Poisoning
2. OIDC Token の奪取

#### 1. GitHub Action の Cache Poisoning

`pull_request_target` イベントをトリガーとしたジョブが実行される。

攻撃を受けた時点での workflow は https://github.com/TanStack/router/blob/d296252f73241c8aca41bd231b57f18e647abd46/.github/workflows/bundle-size.yml から参照可能。今回の攻撃に関係のあるところのみを抜粋すると以下のようになる。

```yaml
name: Bundle Size

on:
  # We use `pull_request_target` to split trust boundaries across jobs:
  # - `benchmark-pr` checks out PR merge code and runs it as untrusted with read-only permissions.
  # - `comment-pr` runs trusted base-repo code with limited write access to upsert the PR comment.
  pull_request_target:
    paths:
      - "packages/**"
      - "benchmarks/**"

permissions:
  contents: read

jobs:
  benchmark-pr:
    name: Benchmark PR
    if: github.event_name == 'pull_request_target'
    runs-on: ubuntu-latest
    outputs:
      current_json_b64: ${{ steps.capture.outputs.current_json_b64 }}
    steps:
      - name: Checkout
      - name: Setup Tools
        uses: TanStack/config/.github/setup@main

      - name: Measure Bundle Size
        run: pnpm nx run @benchmarks/bundle-size:build --outputStyle=stream --skipRemoteCache

      - name: Capture Benchmark Outputs

  comment-pr:
    name: Upsert PR Comment
    if: github.event_name == 'pull_request_target'
    runs-on: ubuntu-latest
    needs: benchmark-pr
    permissions:
      contents: read
      pull-requests: write
    steps:
      - name: Checkout
      - name: Restore Benchmark Outputs
      - name: Read Historical Data (if available)
      - name: Build PR Report
      - name: Upsert Sticky PR Comment
```

コメント部分でも書かれているとおり、PR への書き込み権限はビルドを行う `benchmark-pr` ジョブには与えられていない。`benchmark-pr` ジョブは、PR のコードをチェックアウトしてビルドを行い、結果の一時保持のみを行うようになっている。
`pull_request_target` イベントを利用した workflow であるが、適切に権限を絞って設計されている。

問題 は workflow `TanStack/config/.github/setup` の内側で使われていた `actions/cache` である。

```yaml
- name: Setup pnpm cache
  uses: actions/cache@v5.0.4
  with:
    path: ${{ env.STORE_PATH }}
    key: ${{ runner.os }}-pnpm-store-${{ hashFiles('**/pnpm-lock.yaml') }}-v2
    restore-keys: |
      ${{ runner.os }}-pnpm-store-
```

攻撃者が出した PR はバンドルサイズの計測の過程で、攻撃者のコードは pnpm のキャッシュ領域に悪意のあるコードを仕込ませることができる。
そして、 `actions/cache` は workflow に与えられた権限と無関係にキャッシュをレポジトリスコープで保存してしまう。

この`TanStack/config/.github/setup` は リリースの workflow からも利用されているため、これによって分離された境界を越えて悪意のあるコードをリリースパイプラインに仕込むことに成功している。

なお、`actions/cache` を介した Cache Poisoning 自体は初出ではなく、2025 年時点では報告されていたらしい。

実際にどのようなコードを紛れ込ませたのかの詳細は不明であるが、
リリース用の workflow で利用されるテストなどのジョブから参照される npm パッケージ (e.g. vitest) の中身を書き換えてしまえば、リリース workflow のコンテキストの中で任意のコードが実行できる、ということとなるはず。

#### 2. OIDC Token の奪取

リリースの workflow で汚染されたビルドプロセスからは、以下を用いると npm Trusted Publish に用いられる OIDC Token を参照することができるらしい。

> - Locate the GitHub Actions Runner.Worker process via /proc/\*/cmdline
> - Read /proc/<pid>/maps and /proc/<pid>/mem to dump the worker's memory
> - Extract the OIDC token (which the runner mints lazily, in memory, when id-token: write is set)
> - Use the token to authenticate POST requests directly to registry.npmjs.org — bypassing the workflow's Publish Packages step entirely

正直、 `/proc` を介したメモリダンプについてはあまりイメージが湧いていないが、以下に関連する記事とメモリダンプのコードが記載されていた。

https://0np.de/blog/analyzing-tj-actions-supply-chain-attack/

### 考察

#### 気づきにくい構造

攻撃者の forked PR は極論、一度だけ PR で action が実行されればよいだけ。

- `pull_request_target` イベントの場合、レポジトリオーナー側の承認無しに workflow が実行される
- Cache Poisoning を成立させればよいだけであるため、攻撃者の PR が main に merge される必然性はない
- Cache Poisoning という攻撃特性上、メンテナはその後のリリース作業時にレポジトリが汚染されていることに気づきにくい
  - TanStack の場合、リリースにおけるテストが途中中断されており、それはシグナルの１つであったそうだが、そこはいくらでも凝って偽装できそう

TanStack のポストモーテムにも以下の通り書かれているが、発覚が早期だったことが不幸中の幸い、という印象。

> - External researchers noticed and reported with full technical detail within ~20 min of the incident
> - Maintainer team coordinated immediately and effectively across many timezones
> - The detection community already had a clear public IOC pattern within hours

#### Trusted Publishing をすり抜けている

リリース用権限を持つ workflow が乗っ取られた時点で完全アウト。

例えば 先日に axios が受けた攻撃では、サプライチェーンアタックの入口にはソーシャルハッキングによるメンテナの npm アカウント奪取を経由していた(https://github.com/axios/axios/issues/10636)。
この場合、汚染されたパッケージには Provenance がつかない状態であるので、「来歴を証明できないパッケージは疑う」というスタンスが有効である。

一方、今回の攻撃では正規の OIDC トークンで悪意のあるパッケージの公開に成功している。
TanStack も Trusted Publishing で公開されているが、パッケージ利用者からすると、 Provenance になされた署名は正しい来歴を表していることになる。
npm の Trusted Publishing (Provenance) は SLSA level 3 を保証しているが、利用側からするとそこへの信頼性が揺らぐ、ということになるはず。

axios への攻撃の際は、OSS メンテナ側としては「 Trusted Publishing 以外の経路による Publish を許可しない」という設定を npm で行う、ということで一定の対処ができたが、今回の攻撃手法はそれよりも質が悪い。

#### Trusted Publishing は無意味なのか

https://tanstack.com/blog/incident-followup を読むと、発覚後の追跡の意味合いでは意味があったとのこと。

> With publishes tied back to workflow runs instead of a token, we also had a much easier time understanding the scope of the attack once we had the first report. It showed us exactly what workflow run triggered the publish, which branch it ran from, and when it ran. That audit trail was a significant advantage during incident response and helped us scope the what had happened much more quickly.

確かに、利用者側にとっては改善されたパッケージであっても、トレーサビリティ上の意味はゼロではない、ということか...?

#### CI/CD の権限境界

これも https://tanstack.com/blog/incident-followup に記載されていることではあるが、TanStack はレポジトリのセキュリティ対策を怠っていたわけではない。
前述のように、`pull_request_target` のジョブに対する Permission Control も設計上の意図には問題がない。
GHA の Cache がこの境界を越えてしまう、という穴を知っていて、且つそれを事前に防止しながら CI/CD を組む、というのは不可能に近い要求だと思う。

直接的には、`actions/cache` の暗黙的復元に頼らないようにする、というのは１つの対策ではあるが、「他に workflow の境界を越えてしまうパターンはないのか？」という点については対策にならない。

## Shadow Realm とは何モノなのか

https://css-tricks.com/soon-we-can-finally-banish-javascript-to-the-shadowrealm/ が目に入り、以前に名前だけ覚えていた Shadow Realm の Proposal に改めて思いを馳せた。

言ってしまえば `eval` の代替手段のようなものだ。
`globalThis` を隔離した状態で `eval` ができるメカニズム、というのが一番しっくりくる説明になるかも。

以下は https://github.com/tc39/proposal-shadowrealm/blob/main/explainer.md#quick-api-usage-example に記載されている Shadow Realm API の利用例。

```js
const red = new ShadowRealm();

// realms can import modules that will execute within its own environment.
// When the module is resolved, it captured the binding value, or creates a new
// wrapped function that is connected to the callable binding.
const redAdd = await red.importValue("./inside-code.js", "add");

// redAdd is a wrapped function exotic object that chains its call to the
// respective imported binding.
let result = redAdd(2, 3);

console.assert(result === 5); // yields true

// The evaluate method can provide quick code evaluation within the constructed
// shadowRealm without requiring any module loading, while it still requires CSP
// relaxing.
globalThis.someValue = 1;
red.evaluate("globalThis.someValue = 2"); // Affects only the ShadowRealm's global
console.assert(globalThis.someValue === 1);

// The wrapped functions can also wrap other functions the other way around.
const setUniqueValue = await red.importValue(
  "./inside-code.js",
  "setUniqueValue"
);

/* setUniqueValue = (cb) => (cb(globalThis.someValue) * 2); */

result = setUniqueValue((x) => x ** 3);

console.assert(result === 16); // yields true
```

Shadow Realm API はごくシンプルだ。

- `new ShadowRealm()` で環境を用意
- `importValue` で環境に外側から評価対象をロード
- `evaluate` で実行

Web Worker のように、別の `globalThis` が構成されるわけだが、実行モデルとしては親のメインスレッドがそのまま利用される。

分離されるのはあくまでグローバル変数のみなので、Shadow Realm をもってして「サンドボックス技術」と称するのは無理がある。
ブラウザであれば Web Worker や iframe でより強固な分離が行えるし、Node.js であれば VM Thread のような仕組みが既存で存在する。
JavaScript という言語のレイヤで、`globalThis` のみを切り離した Evaluation がありがたがられるケースというのがあまり思い浮かばない。

https://github.com/tc39/proposal-shadowrealm/blob/main/explainer.md#UseCases にいくつかの例が挙げられている。
強いていうなら、vitest のブラウザモードのようなケースで、テスト対象のコードを Realm に分離して実行する、とかそういうイメージだろうか...?

## 16 Personalities

何年かぶりに https://www.16personalities.com をやったら、INFP-T (仲介者) だった。
何年かに一度ずつ受けるようにしていて、多分今回は 4 回目か 5 回目くらいだと思うが、過去はずっと ENTP(討論者) だったのだが、初めて診断結果が変わったことに驚いている。

おそらく、自分は自分が思っていたほど論理的ではなく共感で動いてることを自覚したのと、こうしてこの雑記を書くような、内省の時間が相対的に増えたあたりが影響してるのかしら。

ちなみに診断結果によると、次のような人間であるらしい。

> 内向的な性格のあなたは、自身の考えや感情、価値観を絶えず探求しようとする傾向があります。この自己認識は自己成長において強力なツールですが、行き過ぎた自己批判につながることもあります。他者からの批判に敏感なので、傷ついたり、誤解されていると感じたりすることもあるでしょう。
>
> このような課題はありますが、自らの理想に揺るぎないコミットメントがあり、他者の長所を見抜く力を備えているあなたは、世の中を良くしてくれる力強い存在です。あなたの自己研鑽プロセスにおいて重要なのは、豊かな内面を大切にしながらも、外界の要求にも適宜応えるように努力することです。そして、常により多くの「美」「優しさ」そして「偽りのないあり方」を、あなたの人生と周りの人の人生にもたらすよう励むといいでしょう。

初めてこの診断をやったとき(社会人 3, 4 年目のときにワークショップの一環だったことを覚えている)、ENTP の診断結果について素直に当たっている、と感じた記憶がある。
今回、上記のコメントを読んで、「まぁそういうところもあるかもな」と思う程度になってしまっている。斜に構える傾向が強くなったのか、感受性がすり減ったのかさてはて。。
