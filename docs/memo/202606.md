## 角丸グラデボーダー

以下の要件を持つデザインに遭遇

- 角丸
- グラデーションボーダー
- 背景透過

どうやって実装したものかと思っていたが、解決方法が https://digipress.info/tech/gradient-border-with-css/ に記載されていた。

```css
.box {
  position: relative;
  padding: 16px;
  background: rgb(255 255 255 / 20%);
  border-radius: 12px;

  &::before {
    position: absolute;
    inset: 0;
    content: "";
    background: linear-gradient(
        135deg,
        rgb(74, 234, 220) 0%,
        rgb(151, 120, 209) 20%,
        rgb(207, 42, 186) 40%,
        rgb(238, 44, 130) 60%,
        rgb(251, 105, 98) 80%,
        rgb(254, 248, 76) 100%
      ) border-box border-box;
    border: 5px solid transparent;
    border-radius: 12px;
    mask: linear-gradient(#000 0 0) padding-box exclude, linear-gradient(
          #000 0 0
        ) border-box exclude;
  }
}
```

ポイントは背景領域を疑似要素で作成し、`background-origin`, `background-clip` でグラデーション背景をボーダー込のサイズに拡張し、それを `mask` で切り取る、という考え方。

# 6/24

## TypeScript の Omit が壊れるとき

仕事にて、`Omit` で合成した型が期待通りに動いておらず、バグを検知できなかったというケースに遭遇した。
`Omit` に関わる部分だけを簡略化すると、以下のような再現コードである。

```ts
type Member = { id: number; name?: string } | undefined;
type Options = Omit<Member, "id">;

// @ts-expect-error
const options = { name: "Bob", age: 20 } satisfies Options;
```

本来は `Member` に存在するもののみを `option` にしたいという意図であり、 `age` のような未定義なプロパティが含まれた時点でエラーにしたいわけだ。
しかし、これが期待通りに動作していなかった。

上記コードでは `Options` として推論される型は `{}` となり、任意のオブジェクトリテラルを通してしまっていた。

これは `Member` の定義に含まれる `| undefined` の部分が問題である。

`Omit` そのものの定義を TypeScript のコードから紐解くと、以下のようになる。

```ts
type Pick<T, K extends keyof T> = {
  [P in K]: T[P];
};

type Exclude<T, U> = T extends U ? never : T;

type Omit<T, K extends keyof any> = Pick<T, Exclude<keyof T, K>>;
```

少し質が悪いのが、`Omit<T, K>` における K は `keyof any` であり、 `keyof T` 以外のものも受け入れてしまう事。

`keyof (X | undefined)` は `keyof X & keyof undefined` であり、 `undefined` に key がないので、元コードでは `keyof Options` が結果的に `never` となる。

仮に `Omit` が `keyof T` のみを受け付けるようになっていれば、 `K` に設定可能な key を書けないので、その時点で気づくことができたはず。

`keyof Options` が `never` であれば、そこから何を Exclude しようが `never` なので、Omit の内側で使われる Pick には `Pick<Member, never>` が渡されることになる。
最終的に `Options` は以下のコードと等価となり、本来欲しかった `{ name?: string } | undefined` ではなく、 `{}` となってしまう。

```ts
type Options = {
  [P in never]: ({ id: number; name?: string } | undefined)[P];
};
```

### 型演算に対する理解

型演算として注意すべきはやはり、 `keyof (X | Y)` が書いた通りには分配されない点だろう。`keyof` の型作用は反変性を持つからだ。

以下のように、T の部分型となる S を考えて、それらの `keyof` を作用した結果を比較してみる。

```ts
type T = { a: string };
type S = T & { b: string };

const keysT = ["a"] as const satisfies (keyof T)[];
const keysS = ["a", "b"] as const satisfies (keyof S)[];
```

`keyof S` に Assignable であっても、 `keyof T` に Assignable ではない。逆に `keyof T` であることが満たされていれば、それは `keyof S` にも Assignable である。

反変性があるということは、 `keyof (X | Y)` は `keyof X & keyof Y` になるということだ。

ところで、 `keyof` と共変性・反変性を扱った記事はあまり例がないようで、日本語だと以下くらいしか見当たらなかった。
https://zenn.dev/line_ec_lea/articles/84a900cda98a13

### 実利用上の話

`Omit` に共用体型を入れると予期しない動きになる、というのが趣旨ではあるが、流石に自分で `| undefined` とか書いた型から `Omit` するような状況はまぁ有りえないだろうと思っている。
実際に起きそうな例でいうと、以下のように紛れ込むパターンではないかしら。

```ts
declare function logInfo(args: {
  message: string;

  // この ?: が undefined との Union を作る
  extra?: {
    trace_id: string;
    categories?: string[];
    clientTimestamp?: number;
  };
}): void;

type Extra = Parameters<typeof logInfo>[0]["extra"];
type OmittedExtra = Omit<Extra, "trace_id">;
```

上記は `logInfo` 引数における `extra` という部分が `?` であるかどうかで結果が変わる。
`OmittedExtra` を定義した時点では意図通りに動作していたとしても、 `logInfo` 関数定義を途中で修正し、 `extra?:` と Optional にした瞬間から `OmittedExtra` は `{}` にすげ変わる。

ぱっと思いつくのは以下あたりか。。いずれも Lint で検知するアプローチ。

- `Omit` を利用させない。 `MyOmit<T, K extends keyof T>  = Omit<T, K>` のようにラップしたうえで、通常の `Omit` の利用をリンタで禁止する
- `Omit` の K が `never` でないことを Type Aware Linting で頑張る

自前 Type Aware Linting は OXLint や Biome 乗り換え考慮すると手を出したくないし、、と思うと前者の方が現実的かしら。
